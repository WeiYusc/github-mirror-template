# github-mirror-template v0.2 简化部署设计稿（中文优先）

> 状态：设计草案  
> 目标：在**不引入高风险一键接管**的前提下，把当前“手工渲染 + 手工审查 + 手工落地”的模板包，推进到“**声明式配置生成** + **中文部署输出** + **可审查 dist 包**”的下一阶段。

---

# 1. 设计目标

v0.2 不追求“自动上线”，而追求三件事：

1. **把零散命令参数收敛成一个声明式配置文件**
2. **把渲染结果整理成标准化部署包**
3. **把部署说明自动生成为中文文档，降低人工拼步骤成本**

一句话概括：

> v0.2 的核心不是 installer，而是 generator。

---

# 2. 明确边界

## 2.1 v0.2 要做的

- 引入 `deploy.yaml` / `deploy.example.yaml`
- 引入新的生成入口，例如：`generate-from-config.sh`
- 基于 YAML 生成：
  - 渲染后的 `conf.d/`
  - 渲染后的 `snippets/`
  - `html/errors/`
  - `RENDERED-VALUES.env`
  - 中文版 `DEPLOY-STEPS.md`
  - 中文版 `DNS-CHECKLIST.md`
  - 中文版 `RISK-NOTES.md`
- 输出一个标准 `dist/<deployment-name>/` 目录
- 保留现有 `render-from-base-domain.sh`，作为底层兼容渲染器

## 2.2 v0.2 明确不做的

- 不自动改线上 Nginx 主配置
- 不自动覆盖宝塔 vhost
- 不自动 reload Nginx
- 不自动申请证书
- 不自动改 DNS
- 不自动探测并修改系统环境
- 不做 Docker Compose 一键上线
- 不做完整宝塔安装器
- 不接管登录、私有仓库、写操作等高风险能力

## 2.3 v0.2 设计原则

- **先生成，再审查，再应用**
- **输出必须可读、可 diff、可回滚**
- **中文优先，但技术标识保持英文**
- **新增能力优先做“生成层”，不直接进入“执行层”**

---

# 3. 当前问题总结

当前模板包已经能用，但部署时仍有几个摩擦点：

1. 输入参数分散在命令行，复用成本高
2. 同一部署信息需要人工重复填写/记录
3. 渲染完成后，部署者还要自己整理 DNS、落地步骤、风险说明
4. 渲染输出存在，但“交付包”概念还不够明确
5. 对第一次使用者来说，阅读成本仍偏高

所以 v0.2 的目标不是改模板逻辑本身，而是：

> 在模板之上补一层“声明式生成器”。

---

# 4. 建议架构分层

建议把项目理解成三层：

## L1：模板层（现有）

已有能力：

- `conf.d/`
- `snippets/`
- `html/errors/`
- `render-from-base-domain.sh`
- `validate-rendered-config.sh`

职责：

- 负责模板本体与底层渲染

## L2：生成层（v0.2 核心）

新增能力：

- `deploy.example.yaml`
- `generate-from-config.sh`
- `scripts/lib/*.sh`（可选）
- 自动生成 `dist/` 部署包
- 自动生成中文部署说明

职责：

- 负责“把用户配置翻译成一个完整可交付部署包”

## L3：应用辅助层（后续版本）

后续可考虑：

- `apply.sh --dry-run`
- `print-targets`
- 部署前检查器
- 环境探测器

职责：

- 负责辅助人工部署，而不是直接代替人工部署

# 5. v0.2 配置模型设计

建议新增：

- `deploy.example.yaml`：公开示例配置
- `deploy.yaml`：用户真实配置（默认不提交）

## 5.1 顶层字段建议

```yaml
deployment_name: github-mirror-prod

domain:
  base_domain: github.example.com
  mode: flat-siblings

tls:
  cert: /etc/ssl/example/fullchain.pem
  key: /etc/ssl/example/privkey.pem

paths:
  error_root: /www/wwwroot/github-mirror-errors
  log_dir: /www/wwwlogs
  output_dir: ./dist/github-mirror-prod

nginx:
  snippets_target_hint: /www/server/nginx/snippets
  vhost_target_hint: /www/server/panel/vhost/nginx
  include_redirect_whitelist_map: true

deployment:
  platform: bt-panel-nginx
  dns_provider: manual
  review_before_apply: true
  generate_checklists: true

docs:
  language: zh-CN
  audience: operator
```

## 5.2 字段设计原则

### deployment_name

作用：

- 用于区分多套部署输出
- 决定 `dist/` 子目录名称
- 方便后续归档与交付

### domain.base_domain

沿用现有核心输入，不改语义。

### domain.mode

沿用现有：

- `nested`
- `flat-siblings`

### tls.cert / tls.key

保留显式路径，不做“自动猜证书”这种高风险行为。

### paths.error_root / paths.log_dir

继续保留当前模板语义。

### paths.output_dir

允许配置最终 `dist/` 目录位置；若不填，则自动派生。

### nginx.*_hint

这里只是“提示目标位置”，**不是自动写入路径**。

这样既能帮用户整理思路，又不意味着脚本会直接改线上目录。

### deployment.platform

v0.2 先只做少数枚举：

- `bt-panel-nginx`
- `plain-nginx`

用途：

- 控制生成的中文说明文档口吻
- 控制部署步骤模板
- 不影响模板核心逻辑

### docs.language

先支持：

- `zh-CN`

后续如果需要可扩展，但 v0.2 先把中文做好。

# 7. 生成流程设计

建议新增入口：

```bash
./generate-from-config.sh --config ./deploy.yaml
```

## 7.1 预期内部流程

### 第一步：读取 YAML

从 `deploy.yaml` 中读取：

- `base_domain`
- `domain mode`
- `cert/key`
- `error_root`
- `log_dir`
- `output_dir`
- 部署平台/文档参数

### 第二步：做轻量校验

校验内容建议仅限：

- 必填字段是否存在
- `mode` 是否合法
- `platform` 是否在支持列表
- 输出目录是否可创建

### 第三步：调用现有底层渲染器

内部转换为：

```bash
./render-from-base-domain.sh \
  --base-domain ... \
  --domain-mode ... \
  --ssl-cert ... \
  --ssl-key ... \
  --error-root ... \
  --log-dir ... \
  --output-dir ...
```

### 第四步：调用现有静态校验器

```bash
./validate-rendered-config.sh --rendered-dir ...
```

### 第五步：补生成中文文档

自动生成：

- `DEPLOY-STEPS.md`
- `DNS-CHECKLIST.md`
- `RISK-NOTES.md`
- 可选：`SUMMARY.md`

### 第六步：打印收尾提示

输出中文摘要，例如：

- 生成完成
- 输出目录位置
- 派生域名列表
- 下一步先做什么

# 8. dist 输出结构设计

建议标准化为：

```text
dist/
└── github-mirror-prod/
    ├── conf.d/
    ├── snippets/
    ├── html/
    │   └── errors/
    ├── RENDERED-VALUES.env
    ├── deploy.resolved.yaml
    ├── DEPLOY-STEPS.md
    ├── DNS-CHECKLIST.md
    ├── RISK-NOTES.md
    └── SUMMARY.md
```

## 8.1 各文件职责

### `conf.d/` / `snippets/` / `html/errors/`

就是现有渲染产物，继续保留。

### `RENDERED-VALUES.env`

保留现有行为，方便排查和人工核对。

### `deploy.resolved.yaml`

建议新增。

作用：

- 把默认值补全后的最终配置保存下来
- 避免后续只看原始 `deploy.yaml` 不知道哪些值是派生的

### `DEPLOY-STEPS.md`

中文部署步骤，面向“实际操作者”。

应该包含：

1. 本次部署目标摘要
2. 本次派生域名清单
3. 需要准备的 DNS 记录
4. 需要放置到哪些目录
5. 宝塔/Nginx 落地顺序
6. `nginx -t` 前检查项
7. 上线后验收项

### `DNS-CHECKLIST.md`

专门把 DNS 拆出来，避免用户在 README 里翻找。

### `RISK-NOTES.md`

写清楚：

- 本包不会自动上线
- 仍需人工审查
- 高危/账号态路径仍是只读受限
- 不能当成 GitHub 完整代理

### `SUMMARY.md`

给一个一页纸摘要，方便快速移交。

# 9. 中文输出文档设计

这是 v0.2 的重点之一。

## 9.1 文风原则

- 默认中文
- 少用项目开发者口吻
- 面向部署操作者
- 不写“也许”“可能应该”这类模糊句
- 尽量直接给清单、路径、顺序

## 9.2 建议输出类型

### DEPLOY-STEPS.md
偏“照着做”

### DNS-CHECKLIST.md
偏“逐项勾选”

### RISK-NOTES.md
偏“边界声明”

### SUMMARY.md
偏“整体总览”

# 10. 推荐实现方式

## 10.1 最小实现方案

v0.2 第一版建议不要引入复杂语言栈。

优先顺序：

1. **Shell + Python（仅解析 YAML）**
2. Shell 拼装中文 markdown 文档

原因：

- 当前项目已经以 shell 为主
- 用户部署环境里通常有 shell
- Python 常见且够用
- 不必为了 v0.2 引入 Node/Ruby/Go 构建链

## 10.2 YAML 解析策略

可接受方案：

- 若系统有 `python3`，用 Python 标准/轻依赖方案解析 YAML
- 若不想依赖 PyYAML，可先将配置模型限制为较窄结构，再用更可控方式解析

更稳一点的现实方案：

- v0.2 文档中明确需要 `python3 + pyyaml`
- 后续再考虑做“无外部依赖版”

## 10.3 不建议的实现方式

当前不建议：

- 先上交互式 wizard
- 先做 TUI
- 先做 Web UI
- 先做 Docker generator
- 先做宝塔自动写配置插件

这些都可以留到后面。

# 11. 与现有脚本的关系

v0.2 不应推翻现有脚本，而应包在它之上。

建议关系如下：

- `render-from-base-domain.sh`
  - 保留
  - 继续做底层渲染器

- `validate-rendered-config.sh`
  - 保留
  - 继续做底层静态检查器

- `generate-from-config.sh`
  - 新增
  - 负责读取 YAML、调用底层脚本、生成中文部署包

也就是说：

> v0.2 是“补一层”，不是“推倒重来”。

# 13. 风险与注意点

## 13.1 最大风险

如果 v0.2 设计失控，最容易走偏成：

- “看起来只是生成器，实际上偷偷开始修改线上文件”

这个必须避免。

## 13.2 配置膨胀风险

如果 `deploy.yaml` 一开始字段太多，会把简单问题复杂化。

所以 v0.2 应坚持：

- 只先覆盖现有核心输入
- 不急着把所有未来想法都塞进去

## 13.3 平台特化过深风险

当前项目虽然偏宝塔，但 v0.2 不应写死成“只能宝塔”。

建议：

- 输出文档可按平台口吻差异化
- 核心配置模型保持通用

再调整 README / INSTALL：

- 加入 v0.2 入口说明

# 15. 当前推荐结论

如果现在要进入实现，最合理的第一步不是直接写完整生成器，而是：

1. 先定 `deploy.example.yaml` 字段集
2. 先定 `dist/` 目录结构
3. 再写 `generate-from-config.sh` 的最小可用版

也就是说：

> 下一步最适合做的是“把配置模型定死”，而不是马上扩张功能面。
