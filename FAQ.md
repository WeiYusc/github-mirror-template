# FAQ.md

# 1. 这是不是完整的 GitHub 镜像？

不是。

它只是 **GitHub 公共资源只读镜像**，目标是让公开页面与下载链路更稳定、更可控，不是复刻完整 GitHub。

---

# 2. 能不能登录？

不能。

登录、OAuth、账号设置、通知、dashboard 这些都不支持。

---

# 3. 能不能访问私有仓库？

不能。

这个项目不接私有仓库，也不代理 GitHub 账号态会话。

---

# 4. 能不能 fork / star / 提 issue / 发 PR？

不能。

这是只读镜像，不支持写操作。

---

# 5. 为什么有些路径会落到“登录已禁用”页面？

因为这些路径属于登录、授权、账号态功能，例如：

- `/login`
- `/session`
- `/signup`
- `/settings/`
- `/account/`
- `/notifications`

这类路径会被单独引导回 GitHub 官方站，而不是继续代理。

---

# 6. 为什么有些路径会落到“只读镜像限制”页面？

因为这些路径对应的是写操作或高风险交互，例如：

- fork
- new issue
- new PR
- compare
- star / watch / sponsor

这类功能不在本项目支持范围内。

---

# 7. 为什么不直接做成“和 GitHub 一模一样”？

因为那会快速滑向：

- 登录代理
- 账号态代理
- 私有资源代理
- 安全边界失控
- 维护复杂度暴涨

所以这套方案刻意停在“公共只读镜像”这条线上。

---

# 8. 为什么 archive / download 不能盲目跟随上游跳转？

因为如果无条件跟随任意 `Location`，会带来：

- 不受控的第三方跳转
- 策略绕过
- 调试困难
- 安全边界变差

所以必须用 redirect whitelist 收口。

---

# 9. 为什么推荐 flat-siblings 域名模型？

因为它更适合现实部署，尤其是：

- 已有 `*.example.com` 通配符证书
- 想复用现有证书
- 想减少嵌套子域层级
- 想在宝塔里按 6 个兄弟域独立建站

---

# 10. 这套东西适合直接一键安装吗？

目前不适合。

现在更适合：

- 渲染模板
- 静态自检
- 人工审查
- 手工部署
- 分步验收

这是为了降低把现网一起带崩的概率。

---

# 11. 宝塔环境一定要一个域名一个站吗？

不一定，但推荐这样做。

因为：

- SSL 管理更清楚
- 日志更清楚
- 回滚更清楚
- 站点职责边界更清楚

---

# 12. 为什么当前阶段不建议关日志？

因为还在收尾和发布前整理阶段。

日志对排查这些问题很关键：

- 某条路径没命中规则
- 某个资源域异常
- 某个上游下载跳转被拦截
- 某个错误页没有正确落地

---

# 13. 如果我要发布这个项目，最少要带哪些文档？

至少建议带：

- `README.md`
- `INSTALL.md`
- `BT-PANEL-DEPLOYMENT-v1.md`
- `DEPLOY-CHECKLIST.md`
- `OPERATIONS.md`
- `FAQ.md`

---

# 14. 如果后面需要支持更多 GitHub 资源域怎么办？

做法应该是：

1. 先确认真实需求
2. 先分析是否属于公共只读范围
3. 先改模板
4. 再同步现网
5. 再补验收与文档

不要直接在现网随手堆规则。
