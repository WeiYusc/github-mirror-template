# FAQ.md

# 1. 这是不是完整的 GitHub 镜像？

不是。

它只是 **GitHub 公共资源只读镜像**，目标是让公开页面与下载链路更稳定、更可控，不是复刻完整 GitHub。

---

# 2. 能不能登录？

不能。

登录、OAuth、账号设置、通知、dashboard 这些都不支持。

---

# 3. 能不能访问私有仓库？

不能。

这个项目不接私有仓库，也不代理 GitHub 账号态会话。

---

# 4. 能不能 fork / star / 提 issue / 发 PR？

不能。

这是只读镜像，不支持写操作。

---

# 5. 为什么有些路径会落到“登录已禁用”页面？

因为这些路径属于登录、授权、账号态功能，例如：

- `/login`
- `/session`
- `/signup`
- `/settings/`
- `/account/`
- `/notifications`

这类路径会被单独引导回 GitHub 官方站，而不是继续代理。

---

# 6. 为什么有些路径会落到“只读镜像限制”页面？

因为这些路径对应的是写操作或高风险交互，例如：

- fork
- new issue
- new PR
- compare
- star / watch / sponsor

这类功能不在本项目支持范围内。

---

# 7. 为什么不直接做成“和 GitHub 一模一样”？

因为那会快速滑向：

- 登录代理
- 账号态代理
- 私有资源代理
- 安全边界失控
- 维护复杂度暴涨

所以这套方案刻意停在“公共只读镜像”这条线上。

---

# 8. 为什么 archive / download 不能盲目跟随上游跳转？

因为如果无条件跟随任意 `Location`，会带来：

- 不受控的第三方跳转
- 策略绕过
- 调试困难
- 安全边界变差

所以必须用 redirect whitelist 收口。

---

# 9. 为什么推荐 flat-siblings 域名模型？

因为它更适合现实部署，尤其是：

- 已有 `*.example.com` 通配符证书
- 想复用现有证书
- 想减少嵌套子域层级
- 想在宝塔里按 6 个兄弟域独立建站

---

# 10. 这套东西适合直接一键安装吗？

目前不适合。

现在更适合：

- 先用 `generate-from-config.sh` 生成部署包
- 或先用 `install-interactive.sh` 做中文交互式收集与受控 apply
- 再人工审查
- 再分步部署
- 再分步验收

这是为了降低把目标环境一起带崩的概率。

---

# 11. 什么时候用 generator，什么时候用 installer？

简单分法：

- 想要**稳定、可版本化、可复用**的配置输入：优先用 `generate-from-config.sh`
- 想要**中文交互收集参数**，并顺手走一次受控 apply / dry-run：可以试 `install-interactive.sh`
- 想直接调最底层模板与渲染参数：用 `render-from-base-domain.sh`

需要注意：当前 `install-interactive.sh` 仍是**实验性编排层**，不是全自动一键安装器。它不会自动改 DNS、不会自动 reload nginx，也不会在失败后自动回滚。

如果没有特别需求，当前默认仍建议优先走：

1. `deploy.example.yaml`
2. `generate-from-config.sh`
3. 人工审查 `dist/<deployment_name>/`
4. 按部署文档落地

因为这条路径最稳定，也最容易做版本管理。

---

# 12. 宝塔环境一定要一个域名一个站吗？

不一定，但推荐这样做。

因为：

- SSL 管理更清楚
- 日志更清楚
- 回滚更清楚
- 站点职责边界更清楚

---

# 13. 为什么通常不建议关日志？

因为日志对排查这些问题很关键：

- 某条路径没命中规则
- 某个资源域异常
- 某个上游下载跳转被拦截
- 某个错误页没有正确落地

---

# 14. 如果我要发布这个项目，最少要带哪些文档？

至少建议带：

- `README.md`
- `INSTALL.md`
- `BT-PANEL-DEPLOYMENT-v1.md`
- `DEPLOY-CHECKLIST.md`
- `OPERATIONS.md`
- `FAQ.md`

---

# 15. 如果后面需要支持更多 GitHub 资源域怎么办？

做法应该是：

1. 先确认真实需求
2. 先分析是否属于公共只读范围
3. 先改模板
4. 再同步到目标环境
5. 再补验收与文档

不要直接在目标环境里随手堆规则。
